# `inputs/` — canonical target definitions

A single source of truth for CASP15, CASP16, and (soon) CASP17 targets,
in a format designed to be trivially **adapted** to any downstream
structure-prediction pipeline (Protenix, chai-lab, Boltz, DynamicBind, etc.)
without per-pipeline re-derivation from the raw CASP bundles.

The idea: **one canonical tree, N pipeline adapters.** Adding a new
prediction tool means writing one adapter in `adapters/`, not forking
the whole input-prep story.

## Layout

```
inputs/
├── README.md                    # this file
├── build_canonical.py           # populate from ../casp{15,16}_ligands/ source data
├── manifest.csv                 # master index, autogenerated
│
├── docs/
│   ├── schema.md                # target.json schema spec
│   ├── conventions.md           # naming, chain IDs, ligand IDs
│   └── adapter_guide.md         # how to write a new pipeline adapter
│
├── casp15/<TARGET>/
│   ├── target.json              # canonical: sequences + ligands + metadata
│   ├── receptor.fasta           # one >entry per chain (redundant with target.json)
│   ├── receptor.pdb             # crystal receptor (protein only)
│   ├── ligands.tsv              # lig_id, ccd, smiles, count (redundant w/ json)
│   └── ligands/
│       └── lig_NN_<CCD>.sdf     # crystal ligand pose(s), one SDF per instance
│
├── casp16/<TARGET>/
│   └── (same layout)
│
├── casp17/<TARGET>/
│   └── (empty until targets release)
│
└── adapters/                    # target.json → pipeline-specific input
    ├── to_protenix_json.py
    ├── to_chai_fasta.py
    └── to_dynamicbind.py        # stub
```

## The `target.json` schema

See [`docs/schema.md`](docs/schema.md) for the formal spec. Quick
example for CASP15 T1124:

```json
{
  "target_id": "T1124",
  "casp_year": 15,
  "set": null,
  "chains": [
    {"id": "A", "type": "protein", "sequence": "NVSL...", "count": 1},
    {"id": "B", "type": "protein", "sequence": "NVSL...", "count": 1}
  ],
  "ligands": [
    {
      "id": "lig_01",
      "ccd": "SAH",
      "smiles": "c1nc(c2c(n1)...)N",
      "count": 2,
      "sdf_paths": ["ligands/lig_01_SAH.sdf", "ligands/lig_02_SAH.sdf"]
    }
  ],
  "receptor_pdb": "receptor.pdb",
  "notes": ""
}
```

Key invariants:

- **Chains**: per-chain sequence + optional count. Identical sequences
  collapse to `count > 1`. Different-length or truncation-variant
  chains stay separate. Never concatenate chains into a fake monomer.
- **Ligands**: grouped by CCD when available; otherwise by a stable
  `lig_NN` ID. `smiles` populated from CCD lookup or source TSV.
- **Paths**: relative to the target directory.

## Running adapters

Each adapter is a standalone script taking `target.json` paths as
inputs:

```bash
# build all CASP15 + CASP16 targets for Protenix
python3 adapters/to_protenix_json.py \
    casp15/*/target.json casp16/*/target.json \
    --out ../casp15_ligands_protenix/jsons/ \
    --out-casp16 ../casp16_ligands_protenix/jsons/

# same for chai
python3 adapters/to_chai_fasta.py \
    casp15/*/target.json casp16/*/target.json \
    --out ../casp15_ligands/fastas/ \
    --out-casp16 ../casp16_ligands/fastas/
```

Adapters should be **one-way transforms** — never modify the canonical
tree, only produce pipeline-specific derivatives.

## Populating from existing data

The CASP15 and CASP16 benchmarks already have derived artifacts
(`fastas/`, `jsons/`, `refs/`) scattered across `../casp{15,16}_ligands*/`.
The `build_canonical.py` script walks these and produces the canonical
inputs tree in-place. Run once at setup:

```bash
python3 build_canonical.py --casp15
python3 build_canonical.py --casp16
python3 build_canonical.py --manifest      # regenerate manifest.csv
```

### CASP17 live ingest

CASP17 targets are pulled live from predictioncenter.org by a separate
spider at `../casp17_ligands/`. Bridge them into the canonical tree:

```bash
python3 build_casp17_from_spider.py            # ingest all current targets
python3 build_casp17_from_spider.py --target T1214
```

CASP17 targets lack crystal ground-truth until organizers release it
post-season, so their `target.json` has `receptor_pdb: ""` and
`sdf_paths: []`. Adapters still work; scoring is blind until post-season.

## Token estimation

Before running inference, estimate memory/GPU needs with:

```bash
python3 adapters/estimate_tokens.py --all --summary
python3 adapters/estimate_tokens.py casp17/T1214/target.json
```

Output includes a GPU tier recommendation per target. Full sizing
guidance lives at `~/data/vaults/docs/GUIDE-gpu-selection-protenix.md`.

Subsequent edits to the canonical tree should go here; the derived
pipelines pull from here via adapters.

## Manifest

`manifest.csv` is the master index — one row per target across all
CASP years. Columns:

```
casp_year,set,target_id,n_chains,n_unique_seqs,n_ligand_entities,n_ligand_copies,primary_ccd,source_receptor_pdb
```

Regenerated automatically whenever `build_canonical.py --manifest` is run.

## What goes here, what doesn't

**Goes here:**
- Canonical target definitions (sequence, SMILES, references).
- Crystal ligand poses (SDF) for scoring.
- Crystal receptor (PDB) for scoring.
- Adapter scripts.

**Does NOT go here:**
- Predicted structures (those live under the per-pipeline directories).
- MSA / template output (those are pipeline-specific caches).
- Scoring CSVs (those live under each pipeline's `out/`).

## For CASP17

When CASP17 targets drop (expected 2026-04-28-ish), the flow is:

1. Drop target definitions into `casp17/<TARGET>/target.json` + receptor/ligand files.
2. Run each adapter to produce pipeline inputs.
3. Each pipeline's own prep + inference + scoring tooling picks up from there.

See [`docs/adapter_guide.md`](docs/adapter_guide.md) for writing a new
adapter for an unfamiliar pipeline.
